## Матрица приоритетов

### P0 — Critical (краши/утечки/поломка сохранений)

**P0-1: Утечка файловых хендлов в загрузке**
- **Локация:** `scripts/scr_saveLoad/scr_saveLoad.gml`
  - файл открывается: строка 15 (`file_text_open_read(save_path)`)
  - ранний `return false` без `file_text_close`: строки 33–36
  - «успешная» ветка тоже не закрывает файл (перед `room_goto`): строки 44–58
  - при исключении `file_text_close` есть (строки 65–67), но это не покрывает нормальный путь
- **Импакт:** накопление открытых файлов → нестабильность, невозможность открыть новые файлы, странные ошибки при долгой игре.
- **Фикс (предложение):** гарантировать `file_text_close(_file)` во всех ветках:
  - закрывать перед каждым `return`
  - либо вынести `file_text_close` в `finally`-паттерн (в GML это обычно `try{...} catch{...} file_text_close(_file);` + флаг успеха).
- **Проверка:**
  - 20–50 повторных загрузок подряд (через меню) без перезапуска игры.
  - загрузка после нескольких быстрых сейвов.

### P1 — High (логика/архитектура, заметные баги, конфликт систем)

**P1-1: `scr_player_ui_blocking` перетирает `can_move` и ломает блокировку катсцены**
- **Локация:** `scripts/scr_player_ui_blocking/scr_player_ui_blocking.gml`
  - вызывает `scr_checkUIBlocking(false, false)` (строка 5) — `include_cutscene=false`
  - при `ui_blocking == false` делает `can_move = true` (строка 11)
- **Импакт:** во время катсцены менеджер выставляет `obj_player.can_move=false`, но `scr_player_ui_blocking` возвращает `false` и может тут же сделать `can_move=true`.
- **Фикс (предложение):**
  - вариант А (минимальный): вызывать `scr_checkUIBlocking(false, true)` (или без второго параметра), чтобы катсцена учитывалась.
  - вариант B (надёжнее): `scr_player_ui_blocking` **не должен принудительно включать** `can_move=true`; только выключать при блокировке UI, а включение — оставлять владельцу (катсцена/диалог/меню).
- **Проверка:**
  - запуск катсцены и попытка двигаться игроком.
  - запуск катсцены → диалог → выход из диалога до завершения катсцены.

**P1-2: Комнатный переход создаёт/уничтожает инстансы в Collision**
- **Локация:** `objects/objRoomChanger/Collision_obj_player.gml`
  - создаёт `obj_changingRoomsController`: строка 14
  - уничтожает `objRoomChanger`: строка 21
- **Импакт:** Collision-ивент может срабатывать в неудобные моменты (особенно при сложной коллизии/нескольких коллайдерах), что усложняет отладку и может давать неочевидные гонки.
- **Фикс (предложение):** перенести создание контроллера в Step (через флаг/очередь), либо использовать «одноразовый» trigger без `instance_destroy` в Collision.
- **Проверка:** пройти через room changer при диагональном движении/скольжении по стене, убедиться что нет дублей контроллера и нет зависаний.

**P1-3: Восстановление управления игроку без учёта катсцены**
- **Локации:**
  - `objects/obj_inGameMenu/Step_0.gml`: строка 49 (`obj_player.can_move = true` при закрытии меню)
  - `objects/obj_settingsManager/Step_0.gml`: строки 176–181 (возврат `can_move=true` в overlay)
  - `scripts/readDialogue/readDialogue.gml`: строка 22 (`obj_player.can_move=false`), а разблокировка происходит в textbox (уже частично защищено)
- **Импакт:** потенциальные конфликты владельцев управления (катсцена/меню/диалог).
- **Фикс (предложение):** унифицировать правило: если активна катсцена (`global.active_cutscene_manager.is_running`) — никто кроме менеджера катсцены не возвращает `can_move=true`.
- **Проверка:** открыть меню/настройки во время катсцены (если возможно), проверить что управление не возвращается раньше времени.
