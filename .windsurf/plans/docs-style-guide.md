# Документ: Стайл-гайд для документации Undefinedtale-888

## Цель
Все `.md` файлы в `docs/` должны использовать максимум возможностей MkDocs Material для читаемости, навигации и визуальной структуры.

## Обязательные элементы (must-have)

### 1. Гиперссылки
- **Внутренние**: использовать относительные пути `../systems/input.md` или `../../architecture/objects.md`
- **Перекрёстные**: каждый раздел, упоминающий другую систему, должен содержать ссылку на неё
- **Anchor-ссылки**: для перехода к конкретным подразделам `#subsection`
- **Не оставлять голые URL** — оборачивать в `[текст](url)`

### 2. Таблицы (tables)
- Использовать для: параметров функций, сравнений, enum-значений, mapping-ов
- Минимум: `| Колонка 1 | Колонка 2 |`
- Для длинных таблиц — использовать header с выравниванием `| :-- | :-- |`

### 3. Admonitions
- `!!! note` — нейтральная заметка, пояснение
- `!!! tip` — рекомендация, best practice
- `!!! warning` — осторожно, side effects
- `!!! danger` — критично, может сломать
- `!!! info` — дополнительная информация
- Заголовок обязателен: `!!! tip "Заголовок"`
- Внутри admonition можно вкладывать код, таблицы, списки

### 4. Code blocks
- Язык обязателен: ` ```gml `, ` ```json `, ` ```yaml `
- Title: ` ```gml title="obj_player/Step_0.gml" `
- Line numbers: ` ```gml linenums="1" `
- Highlight lines: ` ```gml hl_lines="3 5" `
- Code annotations: `// (1)` в коде + `1. пояснение` снаружи

### 5. Content tabs
- Группировать альтернативные примеры (JSON vs GML, разные сценарии)
- Синтаксис:
  ```markdown
  === "Вариант A"
      содержимое
  === "Вариант B"
      содержимое
  ```

### 6. Collapsible blocks (details)
- Для скрытия дополнительной информации:
  ```markdown
  ??? note "Раскрыть подробности"
      скрытый текст
  ```

### 7. Grid cards
- Для landing/overview страниц:
  ```markdown
  <div class="grid cards" markdown>
  - :material-foo: **Заголовок**
    ---
    Описание
    [Подробнее](link)
  </div>
  ```

### 8. Icons (Material Design)
- Префикс `:material-` для material icons
- Префикс `:fontawesome-` для fontawesome brands
- Использовать для визуальных маркеров в списках и карточках

## Стилистика

### Заголовки
- H1 (`#`) — только один на страницу, это title
- H2 (`##`) — основные разделы
- H3 (`###`) — подразделы
- H4+ — только если безальтернативно нужна глубина

### Списки
- Маркированные (`-`) для перечислений
- Нумерованные (`1.`) для последовательностей/порядка действий
- **Жирный текст** в начале элемента списка как "тег" (например: `- **Параметр**: описание`)

### Форматирование
- Имена файлов/функций/переменных — `inline code`
- Пути — `inline code`
- Значения bool/int/string — `inline code`
- Никакого ALL CAPS за исключением аббревиатур

### Язык
- Основной язык документации: русский
- Технические термины: английский (`action_queue`, `actor_map`)
- Имена файлов/функций: как в коде, без перевода

## Структура страницы (рекомендуемая)

```markdown
# Title (H1)

Краткое описание (1-2 предложения)

## Обзор
Что эта страница описывает, зачем нужна

## Основные концепции / Архитектура
Таблицы, схемы, описание

## API / Reference
Функции с параметрами в таблицах, примеры кода

## Примеры
Код с title, line numbers, annotations

## Troubleshooting / FAQ
Admonitions: warning, tip, note

## См. также
Гиперссылки на связанные страницы
```

## Порядок работы (для нейронок)
1. Прочитать файл целиком, определить "вес" (структурность vs сплошной текст)
2. Добавить гиперссылки на все упомянутые системы/файлы/функции
3. Преобразовать сплошные текстовые блоки в таблицы где применимо
4. Добавить admonitions для warnings, tips, notes
5. Обернуть альтернативные примеры в content tabs
6. Добавить code titles/linenums/annotations для блоков кода
7. Добавить "См. также" в конец
8. Проверить: все H2/H3 имеют смысловую нагрузку, нет "висячих" списков
